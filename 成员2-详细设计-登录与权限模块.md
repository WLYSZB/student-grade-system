# 4. 详细设计 — 登录与权限模块

> **负责人：** 成员2  
> **技术栈：** Spring Boot 3.3.5 + MyBatis-Plus 3.5.7 + MySQL 8.0  
> **模块路径：** `controller/auth/`、`service/auth/`、`mapper/`、`dto/`

---

## 4.x.1 模块概述

登录与权限模块是学生成绩管理系统的**安全入口**。系统包含三种角色——系统管理员（ADMIN）、教师（TEACHER）、学生（STUDENT）——不同角色拥有不同的功能访问权限。

### 本模块职责

| 功能 | 说明 |
|------|------|
| **身份认证** | 验证用户名和密码，确认用户身份 |
| **会话管理** | 基于 HttpSession 的登录状态保持 |
| **权限控制** | 配合 LoginInterceptor + RoleInterceptor 实现 RBAC |
| **密码管理** | MD5 加密存储、修改密码、重置密码 |
| **用户注册** | 新用户注册（学生/教师自助注册） |

### 与成员1成果的关系

成员1已完成项目骨架搭建，包括：
- `LoginInterceptor` — 拦截 `/api/**` 请求，检查 Session 中是否有 `currentUser`
- `RoleInterceptor` — 读取请求头 `X-Required-Role`，校验用户角色
- `WebMvcConfig` — 配置拦截器白名单（`/api/auth/login`、`/api/auth/register` 等无需登录）
- `User` 实体类、`ErrorCode` 错误码枚举、`Result<T>` 统一响应格式

本模块在此基础上开发 **AuthController** 和 **AuthService**，实现完整的登录认证业务逻辑。

## 4.x.2 数据库表结构

### sys_user 表（用户表）

成员1设计的 `sys_user` 表结构如下（本模块未修改，仅使用）：

```sql
CREATE TABLE sys_user (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username    VARCHAR(50)  NOT NULL COMMENT '用户名',
    password    VARCHAR(100) NOT NULL COMMENT '密码（MD5加密）',
    role        VARCHAR(20)  NOT NULL COMMENT '角色：ADMIN/TEACHER/STUDENT',
    status      TINYINT      NOT NULL DEFAULT 1 COMMENT '状态：0禁用 1启用',
    create_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted     TINYINT      NOT NULL DEFAULT 0 COMMENT '逻辑删除',
    UNIQUE KEY uk_username (username)
) ENGINE=InnoDB COMMENT='用户表';
```

**初始测试账号：**

| 用户名 | 密码（明文） | 密码（MD5） | 角色 |
|--------|:-----------:|-------------|:----:|
| `admin` | `admin123` | `e64b78fc3bc91bcbc7dc232ba8ec59e0` | ADMIN |
| `teacher1` | `teacher123` | `MD5('teacher123')` | TEACHER |
| `student1` | `student123` | `MD5('student123')` | STUDENT |

> 密码使用 MD5 哈希存储，与成员1的 `init.sql` 中 admin 账号的加密方式保持一致。

## 4.x.3 认证流程设计

### 4.x.3.1 登录认证流程

```
浏览器                          AuthController              AuthService             数据库
  |                                  |                          |                      |
  |-- POST /api/auth/login -------->|                          |                      |
  |   {username, password, role}     |                          |                      |
  |                                  |-- login(username, ------>|                      |
  |                                  |   password, role)        |                      |
  |                                  |                          |-- SELECT * FROM ----->|
  |                                  |                          |   sys_user           |
  |                                  |                          |   WHERE username=?   |
  |                                  |                          |<--- User 对象 -------|
  |                                  |                          |                      |
  |                                  |                          |-- 1. check user!=null|
  |                                  |                          |-- 2. check status=1 |
  |                                  |                          |-- 3. check MD5(pwd) |
  |                                  |                          |-- 4. check role match|
  |                                  |                          |                      |
  |                                  |<-- return User ----------|                      |
  |                                  |                          |                      |
  |                                  |-- session.setAttribute   |                      |
  |                                  |   ("currentUser", user)  |                      |
  |                                  |                          |                      |
  |<--- {code:200, data:{...}} -----|                          |                      |
  |                                  |                          |                      |
  |-- 后续请求 -------------------->|                          |                      |
  |   (Cookie: JSESSIONID)          |                          |                      |
  |                                  |-- LoginInterceptor      |                      |
  |                                  |   getAttribute("curr..")|                      |
  |                                  |   → 有效，放行          |                      |
```

### 4.x.3.2 密码修改流程

```
已登录用户 ──[旧密码 + 新密码]──> AuthController.changePassword()
                                      │
                                      ├─ 1. 从 Session 获取 currentUser
                                      ├─ 2. MD5(旧密码) == 数据库密码？
                                      │     ├─ 不匹配 → 400 "旧密码不正确"
                                      │     └─ 匹配 ↓
                                      ├─ 3. MD5(新密码) == 旧密码？
                                      │     ├─ 相同 → 400 "新密码不能与旧密码相同"
                                      │     └─ 不同 ↓
                                      ├─ 4. 更新数据库密码 = MD5(新密码)
                                      ├─ 5. session.invalidate() 清除登录状态
                                      └─ 6. 返回 200 → 前端跳转登录页
```

### 4.x.3.3 密码重置流程（忘记密码）

```
未登录用户 ──[用户名 + 新密码]──> AuthController.resetPassword()
                                      │
                                      ├─ 1. 参数校验（用户名、新密码非空）
                                      ├─ 2. SELECT * FROM sys_user WHERE username=?
                                      ├─ 3. 用户存在且 status=1？
                                      │     ├─ 否 → 1001 "用户不存在"/1004 "账号已禁用"
                                      │     └─ 是 ↓
                                      ├─ 4. UPDATE sys_user SET password=MD5(新密码)
                                      └─ 5. 返回 200 "密码重置成功"
```

> **安全说明：** 生产环境应在密码重置前发送邮箱验证码，用户输入验证码后方可重置。本课程设计为简化实现，仅通过用户名验证身份。

## 4.x.4 Session 会话设计

### 方案选择

成员1采用 **HttpSession** 而非 JWT 进行会话管理，设计理由：

| 对比项 | HttpSession（方案采用） | JWT |
|--------|:---:|:---:|
| 实现复杂度 | 低（Servlet 容器原生支持） | 中（需自行管理 Token 生成/解析/刷新） |
| 安全性 | 较高（SessionID 随机，可设 HttpOnly） | 依赖密钥保护 |
| 主动失效 | 支持（`session.invalidate()` 立即生效） | 需额外实现黑名单机制 |
| 课程适配性 | ✅ 与 Servlet 教学内容一致 | — |

### Session 数据

| Key | Value 类型 | 说明 |
|-----|:----------:|------|
| `currentUser` | `User` 实体对象 | 当前登录用户完整信息（id, username, role, status） |

### 生命周期

- **创建：** 登录成功时自动创建（`request.getSession()` 由容器管理）
- **失效时机：** ① 用户主动退出（`/api/auth/logout`，调用 `session.invalidate()`）；② 修改密码后自动失效（强制重新登录）；③ Session 超时（Tomcat 默认 30 分钟）
- **校验：** 每次请求由 `LoginInterceptor.preHandle()` 检查 `session.getAttribute("currentUser")`

## 4.x.5 RBAC 权限控制模型

### 架构层次

本系统的权限控制由三层协作完成：

```
┌─────────────────────────────────────┐
│  第1层: LoginInterceptor             │
│  拦截 /api/** 路径（白名单除外）     │
│  检查 Session 中是否有 currentUser   │
│  → 无: 返回 401                     │
│  → 有: 放行到第2层                  │
├─────────────────────────────────────┤
│  第2层: RoleInterceptor             │
│  读取请求头 X-Required-Role         │
│  检查 currentUser.role 是否在列表中  │
│  → 不匹配: 返回 403                 │
│  → 匹配: 放行到 Controller          │
├─────────────────────────────────────┤
│  第3层: Controller 业务层            │
│  执行业务逻辑，返回 Result<T>        │
└─────────────────────────────────────┘
```

### 拦截器白名单

以下接口无需登录即可访问（在 `WebMvcConfig` 中配置）：

| 接口 | 原因 |
|------|------|
| `POST /api/auth/login` | 登录本身不需要已登录 |
| `POST /api/auth/register` | 注册不需要已登录 |
| `POST /api/auth/reset-password` | 忘记密码不需要已登录 |
| `GET /api/notice/list` | 通知公告可公开查看 |
| `GET /api/notice/detail/**` | 通知详情可公开查看 |

### 角色权限矩阵

| 功能模块 | ADMIN | TEACHER | STUDENT |
|----------|:-----:|:-------:|:-------:|
| 登录/注册 | ✓ | ✓ | ✓ |
| 修改密码 | ✓ | ✓ | ✓ |
| 用户管理 | ✓ | ✗ | ✗ |
| 成绩录入 | ✗ | ✓ | ✗ |
| 成绩查询 | ✓ | ✓ | ✓ |
| 通知发布 | ✓ | ✗ | ✗ |

## 4.x.6 统一响应格式

所有 API 使用成员1定义的 `Result<T>` 格式：

**成功响应：**
```json
{
    "code": 200,
    "message": "success",
    "data": {
        "id": 1,
        "username": "admin",
        "role": "ADMIN"
    }
}
```

**错误响应：**
```json
{
    "code": 1001,
    "message": "用户不存在",
    "data": null
}
```

### 本模块使用的错误码

| 错误码 | 含义 | 使用场景 |
|:------:|------|----------|
| 200 | 成功 | 所有正常响应 |
| 400 | 参数错误 | 缺少必填字段、格式不合法 |
| 401 | 未登录 | Session 中无 currentUser |
| 403 | 无权限 | 角色不匹配 |
| 1001 | 用户不存在 | 登录/重置时用户名错误 |
| 1002 | 用户名已存在 | 注册时用户名重复 |
| 1003 | 密码错误 | 登录时密码不匹配 |
| 1004 | 账号已禁用 | 账号被管理员停用 |

## 4.x.9 关于 JWT 修改方案的说明

原（Python Flask 版）方案使用了 JWT Token 认证，但成员1的 Spring Boot 框架已采用 **HttpSession + Cookie** 方案，且 LoginInterceptor 和 RoleInterceptor 已完成。为保持**与成员1架构一致**，本模块最终采用 Session 方案。

**两种方案的对比不影响系统的模块划分**（Controller/Service/Mapper 分层结构一致），接口 URL 和响应格式也与成员1的 README 规范完全一致，其他成员（成员3/4/5）的开发不受影响。
